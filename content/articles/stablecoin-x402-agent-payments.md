# 稳定币与 AI Agent 支付：x402 如何让 API 自主结算

稳定币支付看起来只需要两步：用户转出 USDC，商户收到 USDC。

但真正把它做成产品，还需要回答一整组问题：

```text
这次请求应该付多少钱
允许使用哪条链和哪种资产
谁授权钱包签名
服务端如何验证支付
支付和订单如何关联
重复请求会不会重复扣款
链上成功后业务是否已经履约
失败时如何重试、退款和对账
```

x402 的价值，是把其中一部分能力放进标准 HTTP 请求流程：服务端用 `402 Payment Required` 返回支付要求，客户端完成签名，再携带支付证明重新请求资源。

截至 2026 年 6 月，x402 官方提供 TypeScript、Go 和 Python 客户端与服务端 SDK，并支持 EVM 和 Solana 相关支付路径。它适合 API 按次收费、数字内容付费和 AI Agent 自主购买服务。

# 系统如何运作

x402 的核心流程可以概括为：

```text
客户端请求付费资源
-> 服务端返回 402 和支付要求
-> 客户端钱包构造并签署支付
-> 客户端携带 PAYMENT-SIGNATURE 再次请求
-> 服务端或 facilitator 验证支付
-> facilitator 执行或协助链上结算
-> 服务端返回资源
```

从 HTTP 视角看，它仍然是请求与响应；从钱包视角看，它增加了一次受约束的资产授权；从商户视角看，它需要把支付结果与订单和履约关联起来。

# 402 响应里表达了什么

普通 API 在用户没有权限时可能返回 `401` 或 `403`。x402 使用 `402` 表示：资源存在，但客户端需要先满足支付条件。

支付要求通常需要告诉客户端：

```text
支付金额
资产
网络
收款方
支付方案
资源或请求标识
有效时间
```

客户端不应该根据页面文字猜测金额，而应解析结构化支付要求，确认它符合自己的预算和网络策略后再签名。

# facilitator 为什么存在

商户可以自己实现链上验证和结算，但这意味着要处理不同网络、Token 标准、签名方案、RPC、广播和确认。

facilitator 把这些复杂性收敛成服务接口：

```text
资源服务提交支付载荷
-> facilitator 验证签名和支付条件
-> facilitator 返回验证结果
-> facilitator 执行结算
-> 返回链上交易或结算状态
```

这和钱包后端里的 Chain Adapter 很相似：上层关心“这笔支付是否满足要求”，底层处理具体网络和资产。

但 facilitator 不是魔法。接入第三方 facilitator 意味着商户依赖它的可用性、规则、网络支持和结算正确性；自建则需要自己承担链上基础设施和安全责任。

# 支付成功不等于业务完成

这是支付系统里最重要的状态边界。

区块链结算成功，只能证明资产转移满足链上规则。它不能证明 API 已经返回正确结果，也不能证明数字内容已经交付。

一笔付费 API 请求至少有两条状态链：

```text
支付状态：
created -> signed -> verified -> submitted -> settled

履约状态：
pending -> processing -> delivered / failed
```

二者需要通过稳定的 payment ID 或 order ID 关联。

如果支付已经 settled，但模型调用超时，服务端不能要求用户无条件再付一次。合理做法是让同一订单能够重试履约，或进入退款与人工处理流程。

# 幂等、重放和重复扣款

网络请求可能超时，客户端可能自动重试，Agent 也可能因为没有收到响应而再次调用。

服务端需要确保：

```text
同一支付证明不能无限消费资源
同一订单重复请求不会重复结算
一次支付允许的资源范围是明确的
过期支付要求不能继续使用
请求内容不能在签名后被替换
```

典型幂等键可以由商户订单 ID、资源 ID 和支付载荷共同确定。验证支付之前先查询订单，结算后原子更新支付与履约状态。

# Agent 钱包不是一把无限额私钥

AI Agent 自主支付并不意味着把一个装满资金的私钥交给模型。

Agent 钱包需要策略边界：

```text
单笔最高金额
每日或每周期预算
允许的资产和网络
允许访问的商户或域名
允许购买的资源类型
授权有效期
异常频率限制
人工撤销入口
```

签名服务还应把模型提出的自然语言意图转换成可审计的支付请求：

```text
Agent 决定购买某个 API
-> policy engine 检查预算和白名单
-> wallet 构造受约束支付
-> signer 校验结构化交易
-> 完成签名
-> 记录意图、请求、支付和响应
```

模型可以提出支付意图，但不应绕过策略直接控制私钥。这和 `wallet-api -> wallet-sign` 的安全边界一致：业务层负责决策，签名层只对符合策略的结构化数据签名。

# 商户账务和对账

收到稳定币不等于账务自动完成。

商户需要记录：

- 原始订单金额和计价币种；
- 实际支付资产、数量和网络；
- 支付要求、签名载荷和付款地址；
- facilitator 验证与结算结果；
- 链上交易哈希和确认状态；
- 服务交付结果；
- 退款、争议和费用。

对账至少要比较三份数据：

```text
业务订单
facilitator 结算记录
链上收款地址和 Token 余额
```

如果只依赖 facilitator 回调，可能漏掉链上已结算但通知失败的订单；如果只扫链，又可能无法知道某笔转账购买了哪个资源。

# 价值在哪里捕获

稳定币和 Agent 支付的价值不一定需要由一个概念代币承接。

真实收入可能来自：

- API 或数字内容本身的按次收费；
- facilitator 的验证和结算服务费；
- 商户支付网关和账务软件；
- 钱包托管、策略控制和签名服务；
- 稳定币发行和储备资产收益；
- 多链路由、Gas 代付和兑换服务；
- 风控、身份和合规服务。

判断某个资产是否捕获这个增长，仍然需要检查手续费、股权、治理、回购和强制使用关系。协议使用量增长，不会自动让所有带有“AI 支付”标签的资产获得收入。

# 参与者承担什么风险

## 稳定币风险

稳定币可能出现脱锚、冻结、赎回或发行人风险。商户需要明确接受哪些资产，以及是否自动转换或保留余额。

## 网络和合约风险

链拥堵、RPC 异常、合约漏洞和 Token 授权都会影响支付。多链支持扩大覆盖面，也扩大监控和对账范围。

## facilitator 风险

第三方 facilitator 可能不可用、延迟或返回错误结果。商户应保存验证证据，并为关键业务设计降级和补偿流程。

## 钱包和签名风险

Agent 如果拥有过宽权限，提示注入、错误工具调用或密钥泄露可能直接变成资金损失。

## 履约风险

链上不可逆支付不能自动解决退款和服务质量。用户仍然需要知道服务失败后由谁处理、何时恢复或退款。

# 用 Go 做一个最小实验

一个适合验证 x402 的小产品是“链上地址风险分析 API”。

产品行为：

```text
GET /risk/address/0x...
价格：0.01 USDC / 次
返回：地址标签、交互风险、基础统计和风险说明
```

服务可以拆成：

```text
HTTP API
-> x402 payment middleware
-> order / idempotency store
-> risk analysis service
-> payment ledger
-> chain reconciliation worker
```

一次请求的执行顺序：

```text
1. 客户端请求风险报告
2. 服务端返回 402 支付要求
3. 客户端钱包签署 0.01 USDC 支付
4. facilitator 验证并结算
5. 服务端创建或读取幂等订单
6. 风险服务生成报告
7. 保存履约结果并返回
8. 后台任务核对链上结算
```

Go 版本可以用官方 x402 Go SDK 接入 `net/http` 或 Gin。第一版使用测试网、固定价格和单一资产，先验证支付与履约状态，不急着扩展多链。

# 作为建设者可以做什么

围绕稳定币与 Agent 支付，可以继续发展：

- API 计费、套餐与预算管理；
- Agent 钱包策略和授权控制；
- 多链稳定币支付路由；
- 商户结算、退款和会计导出；
- facilitator 可用性和结算监控；
- 支付风险评分与异常拦截；
- MCP 或 Agent 工具的按次付费市场。

这些方向都需要钱包后端能力：构造支付、隔离签名、追踪交易、维护账务、处理失败，并向用户解释状态。

# 最后理解

x402 不是让 HTTP 自己拥有金钱，而是给 HTTP 请求增加一个标准化的链上支付协商过程。

它解决了“客户端如何知道该付什么、如何提交支付证明、服务端如何验证”的问题，但生产系统还必须补上钱包策略、幂等、履约、退款、账务和对账。

AI Agent 支付真正值得关注的地方，也不是 Agent 会不会自动买东西，而是它能否在明确预算、权限和审计边界内完成一次可信交易。
