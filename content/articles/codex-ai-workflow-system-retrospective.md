# 从重复任务到可恢复工作流：我的 Codex、Skill、Automation 与 MCP 实践

我开始使用 AI 时，最常见的方式是提一个问题，拿到一个答案，然后结束对话。

这种方式对理解概念很有帮助，但它有一个明显限制：下一次遇到相似任务时，我仍然需要重新描述背景、重新整理步骤，也很难知道上一次执行到了哪里。

后来我逐渐把一些反复出现的任务固定下来：

```text
钱包工程验证 loop
公众号增量同步 Skill
Web3 每日研究 Automation
```

这三套系统看起来都在“让 AI 做重复工作”，但它们并不是同一种东西。

钱包工程验证依赖真实服务、数据库、RPC 和链上结果，需要根据反馈持续诊断；公众号同步依赖确定性脚本和持久化状态，AI 只负责摘要判断；每日 Web3 研究则由时间触发，用 Prompt 和 memory 控制选题与输出。

我真正学会的，不只是让 AI 多做一点工作，而是把一个重复任务拆成：

```text
输入
工具
状态
判断
验证
产物
下一次从哪里继续
```

当这些部分被明确下来，一次对话才开始变成工作流。

# 先分清五个概念

在实际使用中，Workflow、Loop、Skill、Automation 和 MCP 很容易被混在一起。我现在会这样区分它们。

| 概念 | 解决的问题 | 在我的系统里的例子 |
| --- | --- | --- |
| Workflow | 一项任务从输入到产物的完整流程 | 公众号文章进入 Obsidian |
| Loop | 根据运行结果继续检查、修复和验证 | 钱包扫链与充值提现测试 |
| Skill | 保存一套可重复调用的方法和工具说明 | 公众号增量同步 Skill |
| Automation | 按时间或规则自动触发任务 | 每日 Web3 巧设计研究 |
| MCP | 把外部工具和数据以受限能力提供给 AI | Obsidian Notes 文件访问 |

## Workflow 是完整业务过程

Workflow 不只是一个 Prompt。它应该明确：输入是什么、按什么顺序执行、调用哪些工具、输出放在哪里、失败后如何恢复。

例如，公众号文章同步并不是“帮我总结一篇文章”，而是：

```text
获取订阅账号文章
-> 分页和清洗
-> 识别待处理文章
-> AI 生成结构化摘要
-> 原子写入 Markdown
-> 更新同步状态
-> 报告成功、跳过和失败数量
```

## Loop 用反馈决定下一步

Loop 适合不能提前写死全部步骤的任务。

钱包服务真实运行时，AI 无法预先知道第一个错误会来自 RPC 字段、数据库约束还是状态机。它只能不断执行：

```text
运行
-> 观察证据
-> 定位原因
-> 最小修复
-> 再次运行
```

直到日志、数据库和链上状态形成一致证据。

## Skill 保存可复用操作方法

Skill 把某类任务的触发条件、步骤、脚本、输出格式和安全边界固定下来。

它的价值不是替模型预先写完所有答案，而是让模型下一次知道应该使用哪套方法，不必由我重新讲解整个流程。

## Automation 负责按时触发

Automation 解决的是“什么时候运行”。它可以携带稳定的任务 Prompt，在指定工作目录中独立执行，并把结果交付给用户。

调度本身不会自动解决选题重复、失败恢复和输出质量，所以仍然需要 memory、状态和明确验收标准。

## MCP 是受权限约束的连接层

MCP 不是让 AI 自动拥有所有软件和接口权限。

它更像一套连接协议：外部系统把允许使用的工具或资源暴露给 AI，AI 只能在被配置的范围内调用。

我当前的 Obsidian MCP 只连接 Notes vault。AI 可以搜索和读取其中的 Markdown，但写入行为仍然受到工具配置和审批约束。它不会因为连接了 Obsidian，就自动获得我的其他账号、插件或任意网络 API 权限。

# 我沉淀下来的统一工作流模型

虽然三套工作流的触发方式不同，但它们可以放进同一个模型：

```text
Trigger
-> Context
-> Deterministic Tools
-> AI Judgment
-> Verification
-> Artifact
-> Memory / State
-> Next Run
```

## Trigger

任务从什么条件开始：我主动提出测试、调用一个 Skill，或者到达每日执行时间。

## Context

AI 需要哪些背景：项目代码、配置、历史笔记、订阅账号、上一轮 memory 或链上交易。

## Deterministic Tools

能由脚本、数据库、RPC、测试命令完成的事情，尽量交给确定性工具。例如分页、去重、写文件、查余额和运行测试。

## AI Judgment

需要语义判断的部分交给模型，例如解释错误、选择最可能的根因、提炼摘要和判断一个协议设计为什么值得研究。

## Verification

不能把模型输出直接当成完成。钱包流程要看日志、数据库和链上状态；内容同步要检查文件存在与 state；研究内容要核对一手来源。

## Artifact

每次运行都要留下可复用产物：代码修复、测试证据、Markdown 笔记、设计卡片或运行报告。

## Memory / State

记录已经处理过什么、失败在哪里、下一次从哪里继续。没有状态，自动化很容易重复劳动。

# 工作流一：钱包工程验证 loop

钱包工程验证是三套系统里最接近真实生产工程的一条。

我的目标不是让 AI 阅读代码后说“充值提现流程看起来没问题”，而是让三套服务真正运行，用一笔真实链上交易验证：

```text
生成充值地址
-> 用户充值
-> wallet-service 扫链
-> Bloom 命中平台地址
-> deposit 入库
-> balance 更新
-> 创建提现
-> 构建交易
-> wallet-sign 签名
-> wallet-api 广播
-> 查询链上结果
```

## 这个 loop 使用了什么上下文

```text
wallet-service / wallet-api / wallet-sign 代码
PostgreSQL 和 Redis
Base RPC
本地服务日志
充值、余额和提现数据表
真实链上交易状态
```

AI 不是只读一个仓库，而是要在代码、运行时和外部事实之间来回核对。

## 第一次运行暴露了什么

三套服务能够启动，不代表扫链闭环已经成立。真实运行依次暴露出三个接口契约问题。

第一个问题是 EVM 区块响应没有回填 `parentHash`。扫链器依赖父哈希验证区块连续性，空值会让系统不断怀疑发生了链重组。

第二个问题是 RPC 返回的交易 `value` 使用十六进制 quantity，而业务层按十进制字符串解析。区块能够被拉取，但交易处理会持续失败重试。

第三个问题是区块 `timestamp` 没有从 RPC 响应传给 service，导致充值命中后无法满足数据库约束。

这三个错误都不是架构图里能直接看到的问题。它们只有在真实链路运行时才会出现：

```text
运行扫链
-> 查看日志
-> 定位停止推进的位置
-> 回到 RPC 数据结构和业务约定
-> 做最小修改
-> 运行测试
-> 重启受影响服务
-> 再次扫同一区块
```

## 如何判断 loop 真正完成

最终验收没有停在“进程没有报错”，而是进行了三层确认：

```text
日志：发现并保存充值交易
数据库：deposit 与 balance 已更新
链上：交易 hash、金额、区块和目标地址一致
```

这种验证方式改变了我使用 AI 的重点。AI 的价值不是替我宣布成功，而是持续缩小问题范围，并把每次修复重新放回完整链路验证。

## 资金操作的边界

钱包工作流涉及真实资产时，Automation 不能等于无限授权。

地址、金额、目标网络和最终广播都应该有明确确认；签名服务也不能因为 AI 发起了请求就绕过风控和审批。AI 可以帮助构建、检查和追踪交易，但资金权限仍然必须由业务策略和人工确认约束。

# 工作流二：公众号增量同步 Skill

公众号文章同步解决的是另一类重复工作：定期获取订阅内容，清洗后沉淀到 Obsidian，并避免重复保存。

截至 2026 年 7 月 3 日，这套 Skill 已经为一个订阅账号沉淀了 20 篇 Markdown 笔记。再次运行时，同一批文章会被 state 识别并跳过。

## 我把确定性工作交给脚本

同步脚本负责：

```text
认证状态检查
账号解析
分页抓取
正文清洗
文章唯一键
重复检测
pending 管理
文件命名
原子写入
state 更新
有限重试
```

这些步骤有明确规则，交给脚本比让模型每次自由发挥更可靠。

## AI 只负责需要判断的部分

模型读取清洗后的正文，生成固定结构的中文摘要：

```text
一句话结论
2-5 个核心观点
我的理解
可执行行动
```

如果原文不支持具体行动，输出空数组，而不是为了填满模板编造建议。

这形成了一条很重要的边界：

```text
脚本负责事实、状态和写入
AI 负责理解、压缩和解释
```

## 失败如何恢复

文章先进入 pending，只有 Markdown 文件成功写入后，才会被标记为 synced。

单篇文章失败不会阻止其他文章；失败项保留在 pending，下一次可以继续。文件名冲突由脚本生成安全名称，已有笔记不会被手工覆盖。

这让同步过程具备可恢复性，而不是一次批处理失败后全部重来。

## 它目前还不是 cron automation

这套系统当前是一项可重复触发的 Skill：当我提出“同步公众号文章”时，Codex 会按固定流程执行。

它已经完成了工作流和状态设计，但还没有被设置成每日定时任务。我选择先验证认证、分页、去重、摘要和写入都足够稳定，再决定是否加调度。

# 工作流三：Web3 每日研究 Automation

第三套系统解决的是长期输入问题：每天研究一个值得学习的 Web3、钱包、跨链或 DeFi 基础设施设计。

它由时间自动触发，输出一张中文设计卡片：

```text
今日项目与一句话妙处
核心机制
为什么巧
如何迁移到钱包三件套
风险或不能照搬的地方
一道复习问题
一手来源
```

## Prompt 里固定了我的工程坐标系

研究结果不能只是协议新闻。每个机制都要映射回：

```text
wallet-service：业务状态和资金状态机
wallet-api：多链 RPC、交易构建、广播和查询
wallet-sign：密钥隔离、签名和签名前校验
```

同时保留一个约束：协议细节可以抽象，但影响资产安全的资源状态必须显式暴露，例如 nonce、UTXO、recent blockhash、object version。

这让每日输入始终服务于我的钱包后端主线。

## memory 解决了什么

定时执行最容易出现的问题是重复选题。Automation 使用 `memory.md` 记录日期、主题、分类和核心结论。

前两次实际选题分别是：

- ERC-4337 的 nonce key / sequence 和 EntryPoint 仿真验证；
- Sui 的 object reference、版本锁和共享对象状态。

下一次运行会读取这些记录，避开同一机制，并在 intent、跨链结算、权限、幂等补偿等方向之间轮换。

所以真正的 Automation 不是：

```text
定时器 + Prompt
```

而是：

```text
调度
+ 稳定任务定义
+ 工作目录
+ 一手来源
+ 历史 memory
+ 固定输出格式
```

# Obsidian MCP 如何成为知识连接层

Obsidian 在这套系统里不是最终归档箱，而是长期上下文。

AI 可以通过 MCP 搜索已有钱包笔记、工作流文档、文章沉淀和索引，再判断新内容应该补到已有题组，还是创建独立原子笔记。

```text
外部文章 / 项目代码 / 链上交易
-> AI 分析
-> Obsidian 笔记
-> MCP 再次读取
-> 后续研究或工程任务复用
```

这形成了知识回流：过去的产物不只供我阅读，也成为下一次 AI 工作的上下文。

但权限边界仍然重要。

我当前只把 Notes vault 暴露给 MCP。读取和搜索在这个目录内进行，写文件仍需要对应工具权限和审批。MCP 负责连接外部能力，不负责替我决定哪些内容应该被修改。

# 可恢复工作流的七条原则

经过这三次实践，我沉淀出七条共同原则。

## 1. 状态必须持久化

钱包用数据库记录区块高度和资金状态；公众号用 state 记录文章；每日研究用 memory 记录选题。

## 2. 重复执行必须安全

重新扫区块不能重复入账，重新同步不能重复建笔记，每日任务不能反复讲同一个机制。

## 3. 确定性工作交给工具

分页、解析、写文件、数据库查询和测试由脚本或工具完成。模型负责工具难以表达的语义判断。

## 4. 失败要隔离

一篇文章失败不能阻塞全部同步；一个区块失败要能从原高度重试；一次研究质量不足不能污染长期 memory。

## 5. 重试必须有边界

不能在认证失败、限流或链上状态不确定时无限循环。错误要被分类、报告，并保留可恢复入口。

## 6. 完成必须有证据

不能只相信 AI 的总结。代码看测试和运行结果，钱包看链上与数据库，内容看文件和 state，研究看一手来源。

## 7. 权限保持最小化

MCP 只暴露需要的目录，Skill 不输出认证信息，Automation 不获得无限资金权限，高风险操作保留人工确认。

# 下一阶段的进步方向

这三套工作流已经能运行，但还可以继续工程化。

第一步是统一运行记录：

```text
run_id
trigger
started_at / finished_at
inputs
result
artifacts
failed_step
next_action
```

第二步是为钱包验证 loop 固定验收清单，至少覆盖服务健康、链上状态、数据库状态、幂等重跑和失败补偿。

第三步是在公众号 Skill 稳定运行一段时间后，再决定是否增加定时调度。调度应该是工作流成熟后的外层，而不是用来掩盖流程不稳定。

第四步是给每日研究增加月度复盘：统计主题分布、重复率、真正迁移到项目的设计数量，并淘汰只停留在“看起来有趣”的内容。

第五步是让以后新增的工作流都回答同一组问题：

```text
什么触发它
需要什么上下文
哪些步骤必须确定性执行
AI 负责什么判断
如何验证完成
状态保存在哪里
失败后从哪里恢复
需要什么权限
```

# 最后的理解

我对 AI 协作的理解，已经从“让模型替我回答问题”变成“让模型参与一个有状态、有工具、有验证的系统”。

钱包 loop 让我学会用真实证据驱动工程修复；公众号 Skill 让我理解确定性脚本和模型判断如何分工；每日 Automation 让我学会用调度和 memory 建立持续输入；Obsidian MCP 则让这些产物能够被后续任务重新查询和组合。

AI 并没有替代我的工程判断。它承担的是重复观察、跨上下文检索、候选原因分析和产物整理，而我仍然负责目标、权限、验收标准和高风险决策。

这套系统真正节省的，不只是执行时间。它开始把一次次对话变成可以积累、可以恢复、也可以继续演进的个人工程能力。
