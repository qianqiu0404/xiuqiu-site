# 问题背景

提现是钱包后端里资金风险最高的一条链路。它看起来像一个简单动作：用户提交提现，系统签名，广播上链，业务方收到结果。但在真实系统里，提现不是一个同步接口调用，而是一条由状态机驱动的异步流程。

这条链路里会遇到节点超时、广播返回丢失、receipt 查询失败、扫链延迟、业务通知失败、账务流水缺失、nonce 或 UTXO 状态不确定等问题。处理这些问题时，最危险的动作不是报错，而是急着“重新发一笔”。

提现错误处理的核心不是“自动还是人工”，而是先判断链上事实是否已经确定。

# 提现是状态机，不是一次请求

一个比较典型的提现状态机可以简化成：

```text
created
-> risk_checking
-> asset_frozen
-> signed
-> broadcasted
-> pending_onchain
-> wallet_done
-> notified
```

每个状态都对应不同的风险边界：

- `created`：用户请求已创建，但资金还没有冻结。
- `risk_checking`：风控正在判断这笔提现是否允许执行。
- `asset_frozen`：用户资产已冻结，防止重复使用同一余额。
- `signed`：签名服务已经生成交易，但不代表交易已经进入链上网络。
- `broadcasted`：系统尝试广播交易，但广播结果可能是确定的，也可能是不确定的。
- `pending_onchain`：交易已被链上网络接收，等待确认。
- `wallet_done`：链上结果已确认，钱包侧可以完成状态。
- `notified`：业务方或上游系统已经收到最终结果。

错误处理必须结合当前状态看。比如 `created` 阶段失败通常不会产生链上风险；但 `broadcasted` 阶段失败就完全不同，因为交易可能已经被节点接收，只是系统没有拿到返回。

# 三层错误处理机制

## 第一层：自动重试

自动重试适合处理“系统请求失败，但没有改变链上事实”的错误。

典型例子包括：

```text
RPC timeout
网络抖动
节点临时不可用
查询 receipt 失败
业务回调通知失败
扫链服务短暂落后
```

这类错误的共同点是：失败发生在通信、查询或通知层，不代表链上交易状态发生了不可控变化。

处理方式可以是：

- 换 RPC 节点重试。
- 延迟重试。
- 指数退避。
- 放入 retry queue。
- 对通知类任务做幂等重放。

自动重试要注意两件事。

第一，重试必须有次数和时间窗口，不能无限重试。第二，重试动作必须幂等。比如业务通知可以重复发，但接收方必须能根据 `withdrawal_id` 或 `tx_hash` 去重。

## 第二层：自动补偿

自动补偿适合处理“链上事实已经确定，但系统状态没有跟上”的错误。

典型例子包括：

```text
链上成功，但订单还停在 broadcasted
链上失败，但订单还显示 pending_onchain
业务通知失败
账务流水没有生成
扫链已经发现交易，但提现单状态没有推进
```

这类问题不是重新发交易，而是把系统状态纠正到链上事实。

常见补偿任务包括：

- 定时扫描 `broadcasted` 或 `pending_onchain` 超时订单。
- 通过 `tx_hash` 查询链上 receipt。
- 如果 receipt `status = 1`，推进到 `wallet_done`。
- 如果链上失败，解冻资产或进入失败处理。
- 如果链上成功但账务流水缺失，补生成账务流水。
- 如果业务通知失败，重新推送通知。

补偿任务的关键是以链上事实和账务事实为准，而不是以某一次接口返回为准。

# 最危险的情况：广播状态不确定

提现错误里最容易出事故的是 `broadcast timeout`。

它看起来像失败：

```text
系统调用 eth_sendRawTransaction
-> RPC timeout
-> 系统没有拿到 txHash
```

但真实情况可能是：

```text
节点已经接收了交易
只是响应丢了
或者当前节点没返回，但交易已经通过网络传播
```

如果这时系统直接重新构建一笔新交易，就可能造成重复出金。

更稳的处理顺序应该是：

```text
先查原 txHash
-> 如果系统保存了 raw_tx，优先重发同一个 raw_tx
-> 多节点查询交易是否存在
-> 检查 EVM nonce 是否已经被占用
-> 检查 BTC UTXO 是否已经被消费
-> 仍不确定时进入人工复核
```

这里的重点是：不要急着构建新交易。

如果同一个 raw transaction 重发，它的 txHash 是确定的，通常不会产生第二笔不同交易；但如果重新构建交易，nonce、gas、UTXO 选择或签名结果变化，就可能变成另一笔交易。

# EVM nonce 的风险

EVM 链上提现通常围绕 nonce 处理交易顺序。nonce 的风险在于：一个地址同一时间只能有一个确定的 nonce 序列。

广播状态不确定时，需要判断：

- 当前 nonce 是否已经被链上消费。
- 相同 nonce 的交易是否已经进入 mempool。
- 是否存在 replacement transaction。
- 新构建交易是否会覆盖旧交易。

如果系统在不确定状态下直接用下一个 nonce 再发一笔，可能出现重复出金。如果用相同 nonce 但不同交易内容，也可能出现替换交易，把原本的交易覆盖成另一笔。

所以 EVM 提现系统通常需要 nonce 管理器，至少保证：

- nonce 分配串行化。
- 同一个提现单绑定确定的 nonce。
- 广播失败后优先追踪原交易，而不是直接新建交易。
- replacement 必须有明确策略和审计记录。

# BTC UTXO 的风险

BTC 不是账户模型，而是 UTXO 模型。提现交易会选择若干 UTXO 作为输入，生成目标输出和找零输出。

广播状态不确定时，需要判断：

- 原交易使用的 UTXO 是否已经被消费。
- 找零输出是否已经生成。
- 是否存在同一批 UTXO 构造出的另一笔交易。
- 是否发生了双花或冲突交易。

如果系统没有确认 UTXO 状态就重新选币构建新交易，可能导致：

- 重复使用同一 UTXO，交易失败。
- 选用新的 UTXO，再发出一笔真实提现。
- 账务系统以为只出了一笔，但链上可能出现多笔相关交易。

所以 BTC 提现错误处理必须把 UTXO 消费状态纳入判断，而不是只看接口超时。

# 账务不平必须人工复核

只要涉及资金账务不一致，就不能简单自动处理。

典型风险包括：

```text
链上金额和订单金额不一致
手续费扣减异常
冻结金额和实际支出不一致
同一订单出现多个 txHash
账务流水重复生成
用户余额和冷热钱包余额对不上
```

这类问题进入人工复核不是因为系统“不智能”，而是因为自动动作可能扩大资金损失。

人工复核应该看到足够上下文：

- 提现单 ID。
- 用户 ID。
- 链和币种。
- 金额和手续费。
- 当前状态机状态。
- txHash / raw_tx。
- nonce 或 UTXO 信息。
- 冻结流水、扣款流水、解冻流水。
- 扫链结果和节点查询结果。

人工处理也不能直接改数据库了事，而应该通过受控的后台动作触发状态推进、账务修正或失败回滚，并留下审计日志。

# 和 wallet-api / wallet-sign 的边界

在钱包系统里，提现错误处理通常不应该全部塞进一个 HTTP handler。

更合理的分工是：

- `wallet-api`：负责提现单状态机、参数校验、风控协作、资产冻结、调用签名服务、广播、状态推进和业务通知。
- `wallet-sign`：负责私钥隔离和交易签名，不应该承担业务状态机。
- 扫链或补偿任务：负责查询链上事实，把链上结果同步回提现单和账务系统。
- 账务系统：负责冻结、扣减、解冻、流水和对账。

这个边界很重要。签名服务只做签名，不判断一笔提现是否应该重试；API 层也不应该因为广播失败就直接调用签名服务重新生成一笔交易。

# 面试回答模板

面试里可以这样回答：

> 提现错误处理不是全靠人工，也不能全部自动重试。我会先按链上事实是否确定来分层。网络抖动、RPC timeout、receipt 查询失败、业务通知失败这类没有改变链上事实的问题，可以通过 retry queue、指数退避和幂等通知自动重试。链上成功但系统状态没更新、账务流水没生成、通知失败这类问题，用补偿任务通过 txHash 查链上 receipt，再推进提现状态和账务流水。但如果是 broadcast timeout、EVM nonce 不确定、BTC UTXO 可能已消费、同一订单多个 txHash、链上金额和订单金额不一致，不能直接重新发一笔，要先查原 txHash、重发同一个 raw_tx、多节点确认、检查 nonce 或 UTXO，仍不确定才进入人工复核，避免重复出金。

这套回答的重点不是背状态名，而是表达出一个工程判断：提现系统的错误处理以资金安全和最终一致性为优先级。

# 常见坑

- 把 timeout 当失败，直接重新构建交易。
- 没有保存 raw transaction，导致广播失败后无法重发同一笔交易。
- 只查一个 RPC 节点，节点查不到就认为交易不存在。
- EVM nonce 没有串行管理，多个提现并发抢 nonce。
- BTC UTXO 没有锁定机制，多个订单选到同一批 UTXO。
- 补偿任务只改订单状态，不处理账务流水。
- 人工后台可以随便改状态，没有审计日志。
