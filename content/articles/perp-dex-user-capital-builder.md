# Perp DEX 参与路径：从交易用户到 Builder

Perp DEX 是普通用户最容易直接体验的 Web3 交易产品之一。

存入保证金、选择交易对、开仓和平仓，整个流程看起来和中心化交易所的永续合约很接近。但把交易搬到链上，并没有消除保证金、Funding、清算和对手方风险，只是改变了订单、资产和状态的保存与验证方式。

真正理解 Perp DEX，需要区分四种角色：

```text
交易用户
资金或 Vault 参与者
平台股权 / 代币持有人
Builder 和工具开发者
```

四种角色面对的收入来源和风险不同。交易量属于协议，不一定属于某个代币；Vault 的历史收益不等于固定收益；Builder 收入则来自自己真正带来的订单和用户。

# 系统如何运作

永续合约没有传统期货的固定到期日。为了让合约价格贴近标的现货价格，系统通常通过 Funding 在多空持仓之间周期性转移资金。

一笔仓位可以抽象为：

```text
用户存入保证金
-> 提交订单
-> 订单进入订单簿或撮合机制
-> 成交后形成仓位
-> 仓位持续计算未实现盈亏与 Funding
-> 保证金率下降到阈值时进入清算
-> 用户平仓或被清算
-> 已实现盈亏回到账户
```

系统至少维护以下状态：

- 账户权益与可用保证金；
- 多空仓位、开仓均价和未实现盈亏；
- 挂单、成交、撤单和拒单；
- 标记价格、预言机价格和 Funding；
- 维持保证金、清算价格和清算记录；
- 手续费、返佣和 Builder fee。

这些状态不是一条简单链上转账能够表达的。对钱包或交易终端来说，展示余额只是入口，真正的产品能力是持续同步账户与订单状态。

# 订单簿、保证金和清算

订单簿描述市场愿意在哪些价格买卖多少资产。用户提交订单后，订单可能处于：

```text
submitted
-> open
-> partially_filled
-> filled
```

也可能进入：

```text
rejected
canceled
expired
```

工程上不能把 HTTP 请求成功等同于订单成交。请求被节点接受、订单进入撮合、部分成交和全部成交是不同状态。

保证金则决定用户能够承受多大的价格波动。杠杆越高，价格只需发生更小幅度的反向变化，就可能使账户低于维持保证金要求。

清算并不是平台随意关闭仓位，而是风险系统在账户权益不足时减少或接管风险敞口。任何交易工具如果只显示潜在收益、不清晰展示清算价格和保证金变化，都没有完整表达产品风险。

# Funding 不是利息账户

Funding 的目标是让永续合约价格靠近现货价格。

当永续价格明显高于现货，多头可能向空头支付 Funding；反过来，则可能由空头向多头支付。具体计算和结算周期取决于协议。

因此 Funding 收益不是稳定利息：

```text
Funding 方向会改变
仓位价格会波动
交易成本会累积
极端行情可能触发清算
```

只看到某段时间的正 Funding，就把持仓理解成固定收益策略，会漏掉价格和杠杆风险。

# 四类参与者分别获得什么

## 交易用户

交易用户通过方向、价差、套利或对冲获得盈亏，同时支付手续费、滑点和 Funding，并承担清算风险。

对新用户来说，最有价值的第一步是以可承受规模跑通入金、下单、撤单、平仓和出金，而不是用高杠杆验证系统。

## Vault 或资金参与者

Vault 可能执行做市、跟单或其他交易策略。资金提供者分享策略盈亏，也承担策略、交易员、合约、流动性和提款限制风险。

判断 Vault 时，需要看：

```text
策略由谁执行
仓位是否透明
最大回撤是多少
提款是否有延迟
收益是否来自承担尾部风险
```

## 平台股权或代币持有人

平台交易量增长只能说明产品被使用。它是否进入某个资产的价值，需要继续检查：

```text
手续费归协议、团队还是用户
是否存在回购或收入分配
代币是否只有治理和激励用途
流通量和解锁如何变化
协议能否在没有该代币的情况下运行
```

代币名称与平台品牌相同，也不能替代这一步。

## Builder

Builder 通过终端、策略、数据、路由或用户体验为协议带来订单。它的价值来自真实使用，而不是被动等待叙事扩张。

截至 2026 年 6 月，Hyperliquid 的 Builder Codes 允许用户授权某个 Builder 的最大费用，Builder 在代用户发送的订单中携带自己的地址和费率，并从实际成交中收取费用。用户可以撤销授权，费用逻辑随订单处理。

这给工具开发者提供了一条清晰路径：

```text
做出用户愿意使用的交易工具
-> 用户明确授权 Builder fee
-> 工具提交带 Builder Code 的订单
-> 订单成交
-> Builder 获得费用
```

# API 工具可以解决什么

Perp DEX 的公开数据和账户接口可以支持很多实际工具：

- 多市场行情与订单簿终端；
- 地址持仓、盈亏和成交分析；
- 清算地图与保证金风险提醒；
- 大额成交和异常 Funding 监控；
- 交易日志、成本和税务统计；
- 经用户授权的策略与跟单工具；
- Builder fee 收入和订单归因看板。

Hyperliquid 的 Info endpoint 可以查询市场和用户状态，WebSocket 适合持续接收行情和账户更新。时间范围接口存在分页和历史数量限制，因此本地服务不能只依赖临时查询，还需要持续消费、落库和补偿。

一个典型的数据链路可以是：

```text
WebSocket 行情 / 用户事件
-> collector
-> 标准化 market / order / fill / position
-> PostgreSQL + Redis
-> 风险计算
-> API / dashboard / alert
```

这和行情服务的工程结构非常接近，但多了账户签名、订单状态和资金风险。

# 交易工具的工程边界

## 签名和 API wallet

交易工具不应该长期保存用户主钱包私钥。协议支持 agent 或 API wallet 时，应使用受限权限完成下单，并把授权、撤销和额度清晰展示给用户。

主钱包负责高权限操作，交易密钥负责有限操作，是比“把私钥交给机器人”更合理的边界。

## 幂等和订单标识

网络超时后，客户端不能不知道订单是否已经提交就盲目重发。需要使用客户端订单 ID、查询订单状态，并对重复请求进行幂等处理。

## 限流和本地缓存

公开 API 存在请求和用户限额。行情、元数据和历史成交应合理缓存，WebSocket 与 REST 查询需要相互补偿。

## 断线恢复

WebSocket 重连后，要根据最后游标或时间戳回补订单与成交，重新校验持仓快照。否则前端看似恢复连接，账户状态可能已经缺失。

## 风险提示

工具必须把杠杆、清算价格、Funding、滑点和订单拒绝原因翻译成用户能理解的状态。只负责“把订单发出去”不等于完成交易产品。

# 参与者承担什么风险

Perp DEX 把市场、协议和操作风险叠加在一起：

- 杠杆和清算风险；
- 标记价格和预言机异常；
- 极端行情下的流动性与滑点；
- 合约、共识或跨链入金风险；
- API wallet 授权和前端供应链风险；
- Vault 策略和提款流动性风险；
- 协议规则、手续费与资产上线变化。

Builder 还多承担一层产品责任：错误订单、陈旧行情、重复提交或错误风险提示都可能直接造成用户资金损失。

# 作为建设者可以做什么

对开发者来说，一个小而真实的切入点可以是“地址交易风险看板”：

```text
输入用户地址
-> 同步仓位、保证金和挂单
-> 计算清算距离与集中风险
-> 展示 Funding 和费用
-> 监听风险阈值
-> 通过邮件、Webhook 或 Telegram 提醒
```

它不需要先做完整交易终端，也不需要替用户保管资金。先解决状态同步和风险解释，就能检验数据质量、用户需求和付费意愿。

# 最后理解

Perp DEX 的核心不只是“把合约交易搬上链”，而是把订单、保证金、Funding、清算和账户状态组成一个持续运行的交易系统。

作为用户，最重要的是理解杠杆和状态；作为资金提供者，要看收益究竟来自什么风险；作为代币持有人，要验证交易收入如何进入资产；作为 Builder，则要为真实订单流提供可靠产品。

对工程师而言，Builder 路径很有吸引力，因为它让产品价值、用户使用和收入之间形成了可以验证的闭环。
